# 用户补充关键点（2026-04-18）

只记录本轮上下文里，用户补充且影响方案方向的关键点。

1. 按顺序推进，先收稳前一层，再进入下一层。
2. 当前阶段先固化流程、规则、模板和脚本，不急着进入具体开发。
3. 先基于前后端分离场景设计整套流程。
4. PRD 拆解前必须先做补充提问，不能直接往下做。
5. 尽量参考已有 skill / 方法论，不完全拍脑袋设计。
6. 双模型先做轻量版：主模型产出，reviewer 审查；后续再增加对抗复杂度。
7. 所有关口都要限制最大 retry，防止磨洋工。当前统一最大 retry = `2`。
8. 固定格式内容尽量脚本化，尽量把 AI 限制在结构化框内。
9. 结构化主格式优先用 YAML，再通过脚本转成给人看的 Markdown。
10. 人工审核后续也应尽量使用固定格式，减少系统漂移。
11. 文档结构要持续收束，核心入口和过程产物分开，避免越做越乱。
12. 初始化流程应独立于 `docs/prd`，单独放到 `docs/init`，不要和普通功能 PRD 流程混在一起。
13. 初始化阶段应先接受用户对整个项目的一句话描述，或直接读取整份 PRD；先分析系统级基座，不先钻具体功能细节。
14. 初始化提问尽量不要做成开放题，优先使用判断题、单选题、多选题；只有确实需要描述时才允许开放题。
15. 初始化提问不是让用户从零填空，AI 应先基于一句话描述 / PRD / 初始参数给出一轮建议值和候选项；最好同时提供 2-3 个常见方案并附带短解释，只有信息明显不足时才允许留空。
16. 专业术语在提问时应补一句白话解释，尤其是 `多租户`、`权限模型`、`账号体系` 这类用户不一定熟悉的概念。
18. 人工确认位置必须在流程里显式可见，不能隐含处理；因此需要明确标出 `Human Confirmation Gate`。
19. `project_profile` 的多阶段不是让 AI 一次性生成四个阶段，而是每个阶段都要先经过人工确认，再进入下一阶段。
20. `project_profile` 四个阶段应固定一组必问题，固定题不能缺；AI 只允许在极少数特殊项目场景下，围绕当前阶段额外补 1-2 个问题，而且不是默认就发散。
22. `baseline` 的核心字段需要支持来源追踪，能够显式追到来自哪个 stage、哪个 `question_id` / `key_decision`。
23. reviewer 不能只做通用审查，还应增加“当前阶段专项检查项”，按阶段逐条检查当前关口应确认的问题。
