# 用户补充关键点（2026-04-18）

只记录本轮上下文里，用户补充且影响方案方向的关键点。

1. 按顺序推进，先收稳前一层，再进入下一层。
2. 当前阶段先固化流程、规则、模板和脚本，不急着进入具体开发。
3. 先基于前后端分离场景设计整套流程。
4. PRD 拆解前必须先做补充提问，不能直接往下做。
5. 尽量参考已有 skill / 方法论，不完全拍脑袋设计。
6. 双模型先做轻量版：主模型产出，reviewer 审查；后续再增加对抗复杂度。
7. 所有关口都要限制最大 retry，防止磨洋工。当前统一最大 retry = `2`。
8. 固定格式内容尽量脚本化，尽量把 AI 限制在结构化框内。
9. 结构化主格式优先用 YAML，再通过脚本转成给人看的 Markdown。
10. 人工审核后续也应尽量使用固定格式，减少系统漂移。
11. 文档结构要持续收束，核心入口和过程产物分开，避免越做越乱。
12. 初始化流程应独立于 `docs/prd`，单独放到 `docs/init`，不要和普通功能 PRD 流程混在一起。
13. 初始化阶段应先接受用户对整个项目的一句话描述，或直接读取整份 PRD；先分析系统级基座，不先钻具体功能细节。
14. 初始化提问尽量不要做成开放题，优先使用判断题、单选题、多选题；只有确实需要描述时才允许开放题。
15. 初始化提问不是让用户从零填空，AI 应先基于一句话描述 / PRD / 初始参数给出一轮建议值和候选项；最好同时提供 2-3 个常见方案并附带短解释，只有信息明显不足时才允许留空。
16. 专业术语在提问时应补一句白话解释，尤其是 `多租户`、`权限模型`、`账号体系` 这类用户不一定熟悉的概念。
18. 人工确认位置必须在流程里显式可见，不能隐含处理；因此需要明确标出 `Human Confirmation Gate`。
19. `project_profile` 的多阶段不是让 AI 一次性生成四个阶段，而是每个阶段都要先经过人工确认，再进入下一阶段。
20. `project_profile` 四个阶段应固定一组必问题，固定题不能缺；AI 只允许在极少数特殊项目场景下，围绕当前阶段额外补 1-2 个问题，而且不是默认就发散。
22. `baseline` 的核心字段需要支持来源追踪，能够显式追到来自哪个 stage、哪个 `question_id` / `key_decision`。
23. reviewer 不能只做通用审查，还应增加“当前阶段专项检查项”，按阶段逐条检查当前关口应确认的问题。
24. `project_profile.md` 不能只是 YAML 的格式化转写，而应明确视为“从结构化 YAML 渲染出来的人工阅读视图层”；更接近“把 JSON 渲染成前端表单/确认卡片”，而不是“把 JSON 漂亮打印一遍”。
25. `project_profile.md` 应聚焦当前阶段，按“项目概览 / 当前阶段结论 / 待确认问题 / 推荐默认项 / 后续阶段预览”这类人读结构输出，而不是把所有 stage 和字段整包平铺给用户。
26. Human Confirmation Gate 对话回复不应机械重复 Markdown 全文；完整内容放在 `rendered/*.md` 给人通读，对话里只拎“真正待确认、且会影响下一步推进”的重点问题。
27. 不是所有固定题都必须再问用户一遍；如果原始材料已经足够明确，AI 应直接采用推荐值，不要为了走流程而重复追问。
28. Human Confirmation Gate 的回复应固定成可直接决策的格式：`问题 + 编号 + 选项 + 推荐项 + 说明`，并允许用户按编号回复修改。
29. 待确认项必须有稳定编号，格式类似 `init-01-01`、`init-02-01`，便于用户用固定格式回复，例如 `编号 init-01-02 改为 [b]`。
30. 用户回复时不能只支持选现有选项；即使给了候选项，也必须允许用户补充自定义答案，例如 `编号 init-01-02 改为 [自定义: xxx]`。
31. 候选项数量不应机械固定为 3 个；原则上控制在 2-5 个，2 个足够表达时直接用 2 个，如果情况太多应拆成多个问题，而不是堆成长选项列表。
32. 需要用户明确答复、且没有安全默认值的问题，不应只落成纯文本开放题；应优先结构化成带 `options`、`recommended`、`allow_multiple`、`allow_custom_answer` 的对象。只有在确实无法再结构化时，才保留开放补充。
33. `open_questions.p0` 这类“必须回复项”不能再只是 `string[]`；后续 `init` 与 `prd` 都应优先采用结构化问题对象，这样脚本渲染层才能给出选项、推荐项和固定回复格式。
34. prompt 优化的核心不是“多写要求”，而是让主模型明确区分三层内容：AI 内部建模用的固定判断、真正需要 human 拍板的关键确认项、以及用户未明确反对时可沿用的推荐默认项，三者不能混写。
35. 后续优化 `prd` 时，应复用本次 `init 1.0.1` 的同一原则：结构化数据层与人工阅读层分离；对话层只问真正不确定的问题；所有确认题尽量编号化、结构化、可脚本渲染。
