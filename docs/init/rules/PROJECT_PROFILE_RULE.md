# 项目画像规则

## 目标

这一步用于把用户输入的一句话描述、项目背景或整份 PRD 先压成一个系统级画像。

这里先不下钻具体功能，而是按阶段递进回答：

- 这是什么系统
- 给谁用
- 第一阶段先确认什么
- 当前阶段确认后，下一阶段该推进什么
- 哪些问题还不能直接进入 baseline

## 基本原则

1. `project_profile` 必须分阶段推进，不再一次性平铺所有初始化问题。
2. 每次只允许 AI 深入当前阶段；已确认阶段保留确认结果，后续阶段保留固定题骨架，但不提前填写正式结论。
3. 先读整体，再问关键基线，不先钻具体页面。
4. 能做成判断题、单选题或多选题的，不要开放提问。
5. 对成熟方案优先给推荐值和备选项。
6. 专业术语首次出现时，要附一句面向非专业用户的解释。
7. 关键决策放进当前阶段的 `key_decisions`，常规默认放进当前阶段的 `recommended_defaults`。
8. 每个阶段的 `required_questions` 是固定题，不能缺漏。
9. `adaptive_questions` 默认应为空，只有确有必要时才允许补 1-2 题。

## 阶段推进规则

每个阶段都必须单独经过下面流程：

1. 主模型产出当前阶段内容
2. 先过脚本校验
3. 再进入 reviewer 审查
4. 人工确认当前阶段
5. 确认后，下一阶段才允许转为 `in_progress`

补充要求：

- 当前阶段之前的阶段必须已经 `confirmed`
- 当前阶段之后的阶段必须保持 `pending`
- 当前阶段的固定题必须完整出现
- 当前阶段固定题原则上都要给出 `recommended`、`options` 和简短 `reason`
- `pending` 阶段不应提前填写正式 `key_decisions`、`recommended_defaults` 或开放问题
- `adaptive_questions` 只能服务于当前阶段，且最多 2 题
- 阶段确认结果写进 `confirmation`
- 只有 4 个阶段全部 `confirmed`，才允许 `allow_baseline: true`

## 推荐阶段划分

建议至少按下面 4 个阶段推进：

1. `foundation_context`
   - 这是什么类型的系统
   - 这个系统主要面向外部客户、外部合作方，还是内部团队
   - 默认地区
   - 默认语言
   - 主要使用端
   - 核心使用场景
2. `tenant_governance`
   - 系统是单租户还是多租户
   - 租户代表的对象是什么
   - 是否存在平台级 / 租户级两层管理
   - 是否需要组织 / 部门结构
   - 组织 / 部门结构主要用于什么
   - 平台侧、租户侧、组织侧的管理边界
3. `identity_access`
   - 默认登录方式
   - 账号唯一标识
   - 账号体系
   - 一个账号能否跨租户
   - 权限模型
   - 是否存在高权限角色
   - 普通成员权限主要按什么分配
4. `experience_platform`
   - UI 风格方向
   - 主题模式
   - 导航方式
   - 信息密度偏好
   - 是否需要通知能力
   - 是否需要审计日志能力
   - 是否需要导出能力
   - 是否需要上传能力
   - 是否需要国际化 / 多语言支持

## 停止推进条件

遇到下面任一情况，不应直接生成初始化基线：

- 当前阶段仍有 `p0`
- `foundation_context` 未完成
- `tenant_governance` 或 `identity_access` 关键结论仍不清楚
- `stage_progress.profile_ready` 还不能置为 `true`

## 对 AI 的要求

AI 在这一步应：

- 提取系统级画像摘要
- 明确当前阶段、已完成阶段和剩余阶段
- 只展开当前阶段，并把后续流程最依赖的问题收敛到当前阶段
- 先基于输入为固定题生成一轮推荐值、候选项和简短理由
- 给推荐选项和推荐默认值
- 避免把大多数问题都写成开放题
- 未进入的阶段只保留固定题骨架与必要空白位，不扩成大题库
- 只有在固定题不足以支撑判断时，才额外补 1-2 个当前阶段问题

AI 不应：

- 直接开始设计普通业务功能
- 直接开始设计页面结构
- 在关键基线问题未确认时擅自拍板复杂架构
- 为后续阶段预先生成大而全的问题清单
- 在当前阶段未人工确认前推进下一阶段
- 把固定题直接变成让用户自由填空
