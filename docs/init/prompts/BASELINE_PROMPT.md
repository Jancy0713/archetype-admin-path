# Baseline Prompt

## 使用方式

把下面 prompt 给主模型，用于在 `init-05 baseline` 阶段对“已确认的 01-04 项目画像”做统一收敛和细节补强。

---

你现在在执行项目初始化流程中的 `init-05 baseline` 步骤。

你的输入不是空白模板，而是：

1. 已完成 01-04 阶段确认的 `project_profile`
2. 一个 `baseline` YAML 骨架
3. 风格参考文档，例如 `UI/UX Pro Max`

你的角色是：

- 主模型
- 负责把 01-04 已确认内容沉淀成后续默认继承的系统基线

你必须遵守以下规则：

1. `baseline` 不是把 01-04 简单摘抄汇总，而是要把已经确认的结论收敛成后续 AI、设计和开发都能直接继承的默认输入。
2. 只能基于已确认的 `project_profile` 输出，不得擅自引入未确认的新系统级结论。
3. 你必须主动补足“当前阶段合理且必要的细节”，不能只保留最抽象的一层描述。
4. 对 `ui_foundation`，不要停留在“专业精致 / 浅色主题”这种口号层；要把风格方向写成后续可执行、可继承的基线描述。
5. 对 `platform_defaults`，不要只写“需要/不需要”；要让后续 AI 能看懂这些能力在初始化阶段意味着什么边界和默认策略。
6. 对 `identity_access`，不要只重复“多租户 / 账号密码 / 轻量 RBAC”；应让后续 PRD 和初始化工程能理解这套基线的落地方向。
7. 每个正式字段都必须保留可追溯来源，优先引用最直接的已确认结构化来源。
8. 如果某些内容仍需人工拍板，才放进 `confirmation_items`；不要为了走流程强行再造问卷。
9. 输出必须严格遵循 `baseline.template.yaml` 字段名。
10. 只输出最终 YAML，不要输出 Markdown 解释。

输出要求：

1. `project_summary` 不仅要说明产品形态，还要体现目标市场、默认地区/语言带来的默认边界。
2. `identity_access` 应足够明确，让后续 AI 知道登录、账号、租户、权限的默认实现方向。
3. `ui_foundation` 应写成“可被 design_seed 继续展开”的基线，而不是口号式摘要。
4. `platform_defaults` 应体现初始化阶段的默认范围和后续继承边界，而不只是 yes/no 标签。
5. `field_sources` 里的 `note` 应让人读得懂“为什么这个值这么定”，而不是只做机械映射。
6. `decision.baseline_confirmed` 默认保持 `false`，除非这是已经完成人工确认后的回填版本。

收敛原则：

- 把 01-04 已确认内容转成“后续默认输入”，而不是“前情回顾”
- 优先补强会影响后续发散度的细节
- 重点让后续 AI 能看懂：什么已经定了，什么不用再重复决策
- 不要把 baseline 写成空泛概括，也不要扩成新的 PRD

请开始。
