# Project Conventions Template

这份模板用于生成项目内长期复用的规则文件。

目标：

1. 把 `init-06 design_seed` 的具体内容展开成后续长期执行的项目规则。
2. 规则文件应长期存在于项目仓库内，而不是只存在于某次 run 的产物中。
3. 写完后，后续 AI 原则上应直接读取这份规则文件，而不是回头再翻 `design_seed`。
4. 每条规则都尽量写成清单项，不写大段散文。
5. 生成方式固定为：脚本先产骨架，AI 再基于当前项目补强和审核。

固定路径：

- 项目规则文件统一落在 `docs/project/project-conventions.md`
- 后续所有 PRD 都默认读取这个相对路径

建议结构：

## 1. 文档定位

- 本文档是当前项目的长期规则来源。
- 后续新建 PRD run、模块开发、review 默认先读取本文档。
- 如需整体修改主题、组件体系、工程基线，进入初始化变更流程，而不是在普通 PRD 中临时重写。

## 2. 设计风格总则

- 固定填写 `design_seed` 已确认的风格方案。
- 固定填写主题模式、暗色扩展策略、信息密度。
- 每项单独列点，不混写成长段描述。

## 3. 导航与应用骨架规则

- 固定填写 app shell 定义。
- 固定填写导航原则。
- 逐条列出页面壳层不得重造的范围。

## 4. Theme / Token 规则

- 分开列出 spacing、radius、shadow、typography、color roles。
- 每一项 token 单独成点，附带用途说明。
- 规则文件应直接展开，不写“详见 design_seed”。

## 5. 页面模式规则

- 每一种页面模式单独成点。
- 写清适用场景、默认结构和不得偏离的重点。

## 6. 组件风格与封装规则

- 每条组件原则单独成点。
- 固定填写优先复用的基础容器与通用组件。
- 固定填写视觉层面禁止出现的偏离方式。

## 7. 禁止事项

- `design_seed.prohibited_patterns` 逐条展开。
- 规则要能被 reviewer 直接检查。

## 8. 使用方式

- 明确 PRD 如何读取本文档。
- 明确设计/开发如何读取本文档。
- 明确 reviewer 如何检查本文档是否被遵守。
