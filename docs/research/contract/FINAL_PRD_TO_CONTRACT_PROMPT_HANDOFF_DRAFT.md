# Final PRD To Contract Prompt Handoff Draft

这份文档用于定义：

- `final_prd` 完成后，系统如何进入 `contract`
- 系统内部应生成哪些 batch 启动文件
- 用户侧应看到哪些信息
- 聊天框与 Markdown 提示词文件如何分工

当前目标不是实现脚本，而是先把交付行为和文件职责定义清楚。

## 目标

在 `final_prd` 通过门禁后：

1. 系统内部根据 `ready_batches` 一次性生成所有 batch 的启动提示词
2. 用户侧只暴露“当前这一轮”应执行的提示词路径
3. 用户若还想调整需求，应回到 `final_prd`
4. 用户若确认进入 `contract`，应按顺序逐轮执行，不提前跳轮

## 为什么这样设计

### 1. 降低用户心智负担

用户不应该在 `final_prd` 之后自己判断：

- 哪个 batch 应先做
- 哪个 batch 可以并行
- 哪个 batch 依赖前置 freeze 结果

这些判断应由系统内部完成。

### 2. 保持 `contract` 的正式边界

每个 batch 都应有：

- 独立上下文
- 独立 run
- 独立 review
- 独立 freeze

如果在同一个长上下文里连续推进多个 batch，边界会很容易混乱。

### 3. 让前序结果自然成为后续输入

推荐的默认执行方式是：

- 先完成 `batch-01`
- `batch-01` freeze 后，作为 `batch-02` 的正式输入参考
- 再推进 `batch-02`
- 再推进 `batch-03`

这比“所有 batch 一起跑 step1，再一起跑 step2”的横切方式稳定很多。

## 用户侧总原则

### 1. 建议新开上下文执行每一轮 contract batch

用户拿到某一轮 batch 启动提示词后，应被明确建议：

- 新开一个上下文执行这一轮
- 不要在 `final_prd` 原上下文里直接继续长链路开发

原因：

- 避免上下文污染
- 让本轮 batch 的输入、review 和 freeze 更清晰
- 降低 token 压力
- 方便失败后单独重跑

### 2. 提示词正文放 Markdown 文件，不直接塞进聊天框

聊天框消息应尽量简洁，只做：

- 告知当前状态
- 告知一共拆成几轮
- 告知当前应执行哪一轮
- 告知当前提示词文件路径
- 告知不要提前执行后续轮次

详细提示词正文应写到 `.md` 文件，由用户按路径打开使用。

原因：

- 提示词通常很长
- 聊天框内直接铺开会干扰用户理解当前重点
- 文件路径更适合复查、复用和后续脚本消费

## 系统内部应生成的文件

假设本次 `final_prd` 共有 `3` 个 ready batches。

在 `final_prd` 完成并确认后，系统内部应一次性生成：

### 1. batch 启动提示词文件

例如：

```text
runs/YYYY-MM-DD-contract-<flow-id>/prompts/
  run-agent-prompt.md
```

每份文件只负责一个 batch 的启动。

### 2. batch 启动索引文件

例如：

```text
runs/<run-id>/contract_handoff/contract-handoff.index.yaml
```

### 3. 用户侧交付说明文件

例如：

```text
runs/<run-id>/contract_handoff/contract-handoff.md
```

这份文件可以用于在需要时完整解释：

- 本次一共拆成多少轮
- 当前轮次是什么
- 后续轮次何时开放
- 如果需要改需求，应该回到哪里

但聊天框里不必完整展开这份文件。

## 聊天框返回的推荐风格

聊天框内容应简洁，不应把整个提示词贴出来。

推荐结构：

1. 当前结论
2. 当前动作
3. 下一步约束

示例：

```md
当前 `final_prd` 已确认，本次一共拆成 3 轮 contract batch。

如果你还要调整需求或边界，我们先回到 `final_prd` 处理。
如果你确认继续，请新开一个上下文执行第 1 轮提示词：
[batch-01.contract-start.md](/absolute/path/to/batch-01.contract-start.md)

请按我给出的顺序推进，不要提前执行第 2/3 轮。第 1 轮完成并确认后，我再继续给你下一轮指引。
```

### 聊天框不应做的事

- 不要直接贴完整提示词正文
- 不要一次性给出所有 batch 的提示词路径
- 不要让用户自己判断依赖顺序
- 不要让用户自己猜是否应该新开上下文

## 当前建议的默认行为

### 1. 所有 batch 提示词一次性生成

这是为了：

- 保持系统状态完整
- 方便脚本和后续调度
- 避免每一轮都重新回头解析 `final_prd`

### 2. 但用户只看到当前 batch

这是为了：

- 控制流程顺序
- 避免越过依赖
- 减少认知负担

### 3. 每轮完成后再显式放行下一轮

建议依赖：

- 当前 batch 已完成 `review_and_freeze`
- 用户确认当前轮结果无误

之后再向用户暴露下一轮提示词。

## 对后续设计的影响

这意味着后续 `contract` 正式流程除了 workflow 自身外，还需要一层“batch 启动交付”能力。

这层能力至少要负责：

1. 从 `final_prd` 解析 ready batches
2. 按顺序生成 batch 启动提示词
3. 维护 batch 启动索引
4. 控制当前向用户暴露哪一轮

## 当前结论

可以先把下面这些点视为已基本确定：

1. `final_prd` 完成后，内部一次性生成所有 batch 的 contract 启动提示词
2. 用户侧只按顺序看到当前一轮的提示词路径
3. 提示词正文写入 Markdown 文件，不直接塞进聊天框
4. 聊天框消息要尽量简洁
5. 每一轮都明确建议用户新开上下文执行
6. 后续轮次必须等前一轮完成并确认后再放行
