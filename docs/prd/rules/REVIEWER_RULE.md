# Reviewer 规则

## 目标

这份文档定义 PRD 拆解流程中的 `Reviewer` 角色。

当前版本采用的是轻量双模型审查机制：

- 主模型负责产出
- Reviewer 负责审查
- 人负责最终判断是否采纳

## 当前定位

Reviewer 不是第二个作者，也不是重新生成一份完整答案的模型。

Reviewer 的职责只有 3 件事：

1. 找遗漏
2. 找矛盾
3. 判断是否允许进入下一步

## 当前适用关口

当前先在 2 个关口启用：

1. `需求补充提问` 结束后
2. `PRD 结构化拆解` 结束后

后续如果流程稳定，再考虑把它扩展到：

- 澄清版 brief
- contract 初稿
- 更重的 debate 机制

## Reviewer 不应该做的事

Reviewer 不应：

- 直接重写主模型正文
- 擅自补全关键业务事实
- 越过当前步骤进入下一步设计
- 为了“看起来更完整”而扩写方案

## Reviewer 应该重点检查的内容

### 在需求补充提问阶段

重点检查：

- 是否漏掉关键问题域
- 是否过早默认业务假设
- 是否把本应让用户选择的关键决策错误地下沉成默认值
- 是否已经把常规问题收束成判断题、单选题或多选题
- 专业术语是否给了足够短的解释，避免用户看不懂问题
- 是否已经足以进入 brief
- 当前 P0 缺口是否已经识别充分

### 在 PRD 结构化拆解阶段

重点检查：

- 模块、页面、角色、资源、流程、状态是否混淆
- 是否把未确认信息写成既定事实
- 待确认项是否独立列出
- 是否还有 P0 级阻塞项未显式指出

## Retry 规则

为了防止流程磨洋工，所有 reviewer 驱动的返工都必须设置最大重试次数。

当前建议：

- 同一关口最大 retry 次数：`2`
- 也就是：
  - 第 1 次主模型产出
  - reviewer 指出问题后最多返工 2 轮
  - 仍未通过则升级给人

超过最大 retry 后，不允许继续无限循环。

## 升级条件

出现以下任一情况时，直接升级给人：

- reviewer 连续 2 轮仍认为存在 P0 问题
- 主模型和 reviewer 对关键边界长期冲突
- 发现文档本身缺失核心输入
- 继续返工只是在换表述，不在解决问题

## 输出要求

Reviewer 输出应尽量短，并使用固定结构：

1. 发现的问题
2. 缺失的信息
3. 是否允许进入下一步
4. 建议返工项

不要输出大段重写版正文。
