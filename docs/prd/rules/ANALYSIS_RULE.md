# 需求分析规则

## 目标

`analysis` 是当前 PRD 2.0 的起点。

它负责：

- 理解输入
- 拆分范围
- 拆出模块 / 页面 / 对象 / 流程初始视图
- 识别风险和阻塞缺口
- 生成进入澄清阶段前的确认候选

## 基本原则

1. 先分析，再提问。
2. 只整理已知信息，不虚构业务事实。
3. 优先识别边界和依赖，不急着写细节方案。
4. 能作为澄清候选的问题，要结构化沉淀，不要散落在正文。

## 正确产出

至少应包含：

1. 输入摘要
2. 范围分析
3. 模块 / 页面 / 资源 / 流程拆分
4. 风险与缺口
5. `clarification_candidates.confirmation_items`

## 不应该做的事

- 不直接代替 clarification
- 不直接代替 execution_plan
- 不把未确认事项写成最终结论
