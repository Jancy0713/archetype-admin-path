# PRD 结构化拆解规则

## 目标

这份文档定义 `拆解 PRD` 的第二个子步骤：PRD 结构化拆解。

它的目标不是补充需求，而是把已经完成补问的 `澄清版 brief` 转成后续步骤可以稳定消费的结构化结果。

## 这一步在整体流程中的位置

当前顺序固定为：

1. 需求接收
2. 需求补充提问
3. 澄清版 brief
4. PRD 结构化拆解
5. contract 设计

如果没有 `澄清版 brief`，原则上不进入这一步。

## 这一步要解决什么问题

自然语言 PRD 即使已经很详细，仍然会存在这些问题：

- 信息分散在不同段落和页面描述中
- 页面、资源、角色和流程交叉混写
- 状态流转只隐含存在，没有单独被整理出来
- 待确认项混在正文里，后续容易被忽略

结构化拆解的作用，就是把这些信息重新整理成固定视角。

## 正确产出

这一步的输出至少要包含：

1. 模块清单
2. 页面清单
3. 角色清单
4. 资源清单
5. 核心流程清单
6. 状态清单
7. 权限与租户观察
8. 待确认项

## 基本原则

1. 只整理已知信息，不虚构业务细节。
2. 可以做合理归类，但不能把尚未确认的内容写成既定事实。
3. 待确认项必须独立列出，不能埋在结构正文里。
4. 页面、资源、流程、状态必须分开整理，不能混在一个段落中。
5. 结构化拆解是后续 contract 的输入，不是 contract 本身。

## 推荐拆解维度

### 1. 模块清单

按系统级模块列出，例如：

- 商家端
- 运营后台
- 管理后台
- AI 视频生成引擎
- 合规引擎

### 2. 页面清单

按端拆出页面或视图入口，例如：

- 一级页面
- 二级页面
- 配置页
- 列表页
- 详情页

### 3. 角色清单

列出当前已知角色，并标记：

- 所属端
- 主要目标
- 当前已知权限特征

### 4. 资源清单

资源应优先围绕业务对象，而不是围绕页面控件。

例如：

- 租户
- 商家账号
- 视频生成任务
- 素材
- 合规报告
- 积分流水

### 5. 核心流程清单

每条流程至少整理：

- 触发方
- 起点
- 关键步骤
- 终点
- 是否异步

### 6. 状态清单

即使状态还不完整，也要尽量显式整理：

- 哪些资源明确有状态
- 当前文档里已经出现了哪些状态或阶段
- 哪些状态仍需后续确认

### 7. 权限与租户观察

这一部分先做观察，不急着设计完整权限模型。

至少要整理：

- 当前文档显式提到的权限信息
- 当前能看出的租户隔离迹象
- 明显缺失的边界

### 8. 待确认项

待确认项要按优先级整理：

- P0：不确认不能进入 contract
- P1：不确认会影响 contract 细节
- P2：可后续继续细化

## 不应该做的事

这一步不应该：

- 直接设计接口
- 直接设计数据库
- 直接设计组件树
- 直接补全完整权限矩阵

这些内容属于后续步骤。

## 和下一步的关系

结构化拆解完成后，下一步才能开始设计 contract。

因为 contract 依赖的不是原始 PRD，而是：

- 经补问后的 brief
- 经结构化拆解后的稳定对象视图
