## 说明

- 来源：小红书文章 OCR 整理稿
- 当前版本：基于原始转文字内容做了人工清洗和排版

## 背景

作者在一年前写过一篇用 AI Coding 从 0 到 1 搭建可控产品 Demo 的文章。那时 `vibe coding` 的概念刚出来，作者也比较早在组织内做了实践。核心方法一直没变：

- 用 AI 快速搭建实验
- 在探索出确定性结论后，尽快固化下来
- 再进入更正式的研发流程

这一年里，随着 Claude Code 一类 coding agent 的能力提升，很多人开始强调“让 agent 7x24 不间断工作”。作者在尝试后发现，真正的关键矛盾并不是“能不能一直跑”，而是：

- 执行长度
- 执行深度

如果只想让模型一直工作，最简单的方法是反复喂重复任务，用 hook 或 loop 不断触发执行循环。但产品经理的典型场景并不是这样。更多时候，PM 需要做的是：

- 定义当前阶段真正要解决的问题
- 讨论理想态
- 找出现状和理想态之间的 gap
- 分析关键约束
- 研究解题策略

如果这些高密度思考环节被省略，最终产出的产品文档就会失控，coding agent 会像一匹失控的野马在代码库里狂奔。

作者对 `harness` 的理解是：

> harness 的本质，是想尽办法控制住 coding agent 这匹快马，让它按你的意愿前进。

经过一段时间实践，作者形成了一套自己的 harness 方法论，并在内部持续运转。它不追求夸张的“全天候全自动”，而是追求：

- 在产品设计研发流程中
- 持续数小时地
- 高质量自动化推进工作

## 一、做了什么

作者设计了一套从问题定义到产品落地的半自动流水线，其中最重要的有两部分：

1. `Debate` 机制：中高质量 PRD 的流水线
2. `Conductor` 机制：需求落地总管

---

## 1. Debate 机制：中高质量 PRD 的流水线

作者一开始是手动一步步和 LLM 沟通，讨论出高质量 PRD，再推动落地。随着实践越来越熟练，提问方式逐渐可以固化成一个稳定框架：

- 界定 / 定义问题
- 讨论理想态
- 讨论 gap
- 分析关键约束
- 研究解题策略
- 权衡并决定路线
- 讨论方案

后来作者发现，自己很多时候只需要做一件事：

- 判断当前阶段的话题是否已经讨论充分
- 如果充分了，就进入下一层

再往后，作者尝试把“是否推进”的判断也外包给模型：

- 一个模型负责分析
- 另一个模型负责 review
- 由人做观察和转发

再后来，连“转发工具人”这个角色也被 agent 接管，于是形成了 `Debate` 机制。

### Debate 的基本结构

- 用户给出主题或问题
- 一个 agent 作为 host
- host 负责组织讨论，以产出最合理的需求文档为目标
- 在真正开辩之前，先帮用户把问题定义清楚
- 随后由两个 sub-agent 用不同材料、不同视角进行对抗
  - proposer
  - reviewer

用户可以参与，也可以暂时离开。通常 1 到 2 小时、十几轮讨论之后，就能交付出一份高质量的讨论结论和 PRD。

### 实例：不是直接修 6 个问题，而是先上升到结构层

作者举了一个真实例子：评测平台里有一个嵌入式 AI 对话面板，帮助用户配置评测项目。某个版本上线后发现了 6 个体验问题。

如果直接让 AI “解决这 6 个问题”，往往只会得到 6 个局部修补方案，头痛医头、脚痛医脚。

于是作者发起了一次 Debate。讨论跑了 14 轮，逐层推进了四个层级：

1. 问题定义
2. 理想态
3. 差距分析
4. 策略

其中最精彩的一段发生在第二层。某个模型提出了用户旅程和 5 个产品属性，另一个模型立刻指出：

- 这还只是能力描述
- 还缺少几个更基础的定义

例如：

- 工作台与主系统的关系
- Session 生命周期
- 工作材料对象模型

这次 review 的价值在于：

- 它把讨论从“怎么做”拉回到了“产品结构到底是什么”

### Debate 里的一个关键机制：回溯校验

在策略层裁剪 scope 时，Debate 里还有一个重要机制，叫“回溯校验”。

做法是：

- 每次裁剪方案
- host 都要求 agent 把当前 scope 逐条映射回早期确认的核心承诺
- 检查有没有被偷偷架空

它解决的是一个很常见的问题：

- 讨论往后走，前面说过的话就失效了

通过这次讨论，最终产出了四层共识文档：

- 产品定位
- 理想态
- 四维度差距分析
- 三段式分期策略

作者认为，这份文档质量显著高于单个 super-agent 在单 session 里直接输出的结果。

### 从截图中能看出的几个关键思想

根据截图内容，可以提炼出几条非常重要的方法：

#### 1. 进入差距分析前，理想态必须“收全”

不是有几个功能点就够了，而是要把对象、边界、生命周期、材料模型等更底层定义补齐。

#### 2. 差距分析要按维度，不要回到零散问题清单

截图中提到的维度包括：

- 对象与边界
- 生命周期
- 工作可见性与控制权
- 信息架构

这意味着：

- Layer 3 不应该只是把 PM 发现的问题重新列一遍
- 而应该把现状和理想态进行系统性对照

#### 3. 不是所有“小改动”都适合先做

作者提出了 `no-regret` 改进的概念。

只有同时满足下面两个条件的改动，才值得提前做：

- 不依赖底层工作单元能力
- 底层能力补齐后，这个改动仍然成立，不会返工

据此，策略主线不是“先小后大”，而是：

1. 先收割 no-regret 改进
2. 再建设工作单元底座
3. 最后整合工作流和信息架构

## 2. Conductor 机制：需求落地总管

即便 Debate 阶段产出了高质量产品文档，落地阶段仍然要非常小心。作者特别警惕 agent 那种“大包大揽、一口气做完”的倾向。

作者的判断是：

- 如果不强加约束，多数 agent 喜欢一口气把问题解决
- 但通常解决得越快，质量越低

作者自己不懂代码，因此在落地时更加谨慎，传统研发中的质量环节几乎一个都不敢省：

- 方案设计
- 架构评审
- UX 评审
- 方案拆解
- 开发
- 增量审查
- 架构 review
- 测试

后来，作者把这些环节设计成了一套流水线，也就是 `Conductor`。

### Conductor 的角色

Conductor 是一个状态机驱动的调度中枢，它自己不写代码、不做设计，只负责：

- 按 pipeline 状态调度专门化 worker

例如：

- Architect：做方案
- Dev：写代码
- UX Evaluator：做体验审查
- Reviewer：出评分卡

### 最关键的设计：增量执行循环

Conductor 最核心的机制不是“大需求一次做完”，而是：

- 按任务清单逐个执行
- 每个 task 做完就审查
- 有问题当场修
- 通过后再进入下一个 task

典型流程是：

1. Dev 完成一个 task
2. 必须连同测试结果一起交付
3. 立刻审查这个 task 的代码变更
4. 如果有问题，直接让同一个 Dev 修
5. 通过后，下一个 task 再交给新的 agent

作者强调：

- 同一个 task 内保留上下文，有利于快速修复
- 不同 task 使用新 agent，可以保持上下文干净

这个设计解决的是：

- 错误越早发现，返工越便宜

如果 task-1 引入了错误模式，却等到 task-3 全部做完再一起 review，那么返工范围就不再是一个 task，而可能是整个需求。

### 对 context window 的处理思路

作者没有试图靠压缩上下文去“续命”，而是干脆从流程设计上规避上下文过长：

- 每个 task 用新的 agent 启动
- 通过 `progress.md` 做结构化交接

这样每个 Dev 面对的都是干净上下文，而不是越来越臃肿的会话。

### 分级升级协议

作者把这套机制称为“半自动”，因为它有明确的升级规则：

- 方案涉及用户可见变更或数据库迁移：暂停，等待人工确认
- 审查出现 blocker 级问题：先自动返工，最多 2 轮，再升级给人工
- 验收阶段：无条件暂停，不能绕过人的最终确认

于是，作者自己的日常变成了：

- 定义问题
- 通过 Debate 做辩证展开
- 审阅方案
- 确认推进
- 最终验收

其余部分交给 agent 流水线协作完成。

## 二、作者对 Harness 的理解

作者认为，agent 单次执行的质量，主要取决于以下几点：

### 1. 任务难度是否在能力圈内

如果给 agent 的任务本身就不适合当前能力，例如“规划一个凭空捏造的 Super App”，它再怎么拆解也难落地。

### 2. 是否正确引导 agent 拆解任务

LLM 的思维空间理论上几乎是无限的，这既是优势，也是风险：

- 如果引导得好，它可能做出比多数人类更高水平的输出
- 如果不引导，它的注意力不一定落在你真正关心的地方

### 3. 草稿纸是否足够好

这里的“草稿纸”，就是文档系统和 context 传递结构。

如果任务能被拆成理想粒度，那么每个 ReAct loop agent 理论上都能做好自己的工作。剩下的问题就是：

- 草稿纸够不够长
- 草稿纸设计得够不够合理

作者认为，上面这三点共同构成了 context engineering 需要解决的问题。

### 4. Harness 是再往上的一层

在作者看来，harness 比 context engineering 多了一层：

> 怎么设计流程机制，让 agent 自己在机制下跑起来，并且是按 1 到 3 的最佳实践去跑。

也就是说，harness 不是单条 prompt，而是控制 agent 前进方式的“马具”。

## 对我们最有参考价值的部分

如果只提炼和我们当前工作最相关的结论，这篇文章最值得吸收的是：

1. PRD 不是一轮对话直接写出来的，而是分层推进出来的
2. “是否允许进入下一层”本身应该成为机制的一部分
3. 理想态、gap、策略必须分层，不要混在一个 prompt 里
4. 需求落地要做 task 粒度的审查闭环
5. 流程和文档系统是 harness 的核心，不是单个聪明 prompt
