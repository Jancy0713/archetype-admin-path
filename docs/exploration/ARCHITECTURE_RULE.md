# Archetype Admin 结构约束

## 目标

这份文档用于固定项目在探索阶段的目录结构原则，避免 AI 或人工在每次新增功能时重新发明一套组织方式。

当前目标不是一次性设计完整系统，而是先建立一套可以持续扩展的结构契约。

## 当前结论

1. 要统一目录结构，但不要把所有类型、逻辑、页面塞进少数几个总文件。
2. 要有 schema 作为真相源，但不采用单个无限膨胀的 `src/types/schema.ts`。
3. 结构统一的重点是“按资源拆分”，而不是“按文件类型全局堆叠”。
4. 先做一个可以复制的黄金切片，再逐步扩展到更多资源。

## 反对的结构

以下结构当前不建议采用：

- 所有实体都放进一个总 `schema.ts`
- 所有 API 映射都堆进一个全局 `dataProvider.ts` 文件中直接硬编码业务细节
- 所有页面都直接放在 `src/pages` 下平铺增长
- 页面内部自己定义类型、请求和字段映射

这些做法在项目早期看起来快，但会让后续 AI 修改越来越不稳定。

## 当前推荐结构

推荐先按下面的层次组织：

```txt
src/
  app/
    providers/
    router/
  shared/
    ui/
    lib/
    types/
  features/
    users/
      schema/
        entity.ts
        form.ts
        query.ts
      api/
        mapper.ts
      components/
      pages/
        list.tsx
        create.tsx
        edit.tsx
```

## 每一层的职责

### `app/`

用于放应用级装配逻辑：

- Refine 根配置
- providers 注册
- resources 注册
- 路由入口

这里不放具体业务字段定义。

### `shared/`

用于放跨资源复用能力：

- `shared/ui`：通用 UI 包装或布局组件
- `shared/lib`：工具函数
- `shared/types`：真正跨资源共享的基础类型

只有在多个资源都依赖的前提下，类型才允许进入 `shared/types`。

### `features/<resource>/`

每个资源独立维护自己的结构，是当前阶段最重要的组织单位。

以 `users` 为例：

- `schema/entity.ts`：资源实体结构
- `schema/form.ts`：表单输入结构
- `schema/query.ts`：筛选、排序、查询参数结构
- `api/mapper.ts`：后端字段与前端字段之间的映射
- `components/`：仅供该资源复用的视图组件
- `pages/`：list/create/edit/show 等页面

## 关于 schema 的结论

当前明确不采用一个全局总 `schema.ts`。

原因：

1. 随着资源增长，这个文件会快速失控
2. 不同资源的变更会互相干扰
3. AI 在超大单文件里更容易做局部兼容和脏修补
4. 后续抽离开源模板时也不利于模块化

当前采用的原则是：

- schema 必须统一存在
- 但 schema 统一的是“约定”，不是“单文件”

也就是：

- 每个资源都有自己的 schema 目录
- 所有页面和映射都优先依赖该资源下的 schema
- 真正跨资源的基础类型再进入 `shared/types`

## 关于 Refine 的边界

当前对 Refine 的使用边界这样约束：

- `resources` 负责声明模块和页面挂载关系
- `dataProvider` 负责与后端协议对接
- `hooks` 负责列表、表单、交互状态调度
- `shadcn/ui` 组件只承担展示

不要把页面写成“大一统组件”，同时处理数据、映射、类型和展示。

## 黄金切片原则

当前项目不是要先做“大而全”的架构，而是先做一个最小但标准的资源样板。

推荐第一个黄金切片资源：`users`

原因：

- 用户资源最容易覆盖列表、表单、状态字段、认证关联等典型后台问题
- 后续无论做订单、内容、配置，都会复用这套组织方式

黄金切片的目标不是做完业务，而是建立之后所有资源都能复制的标准。

## 后续演进规则

后续新增资源时，优先复制黄金切片目录，而不是重新设计结构。

允许演进，但必须遵守：

1. 先在文档里说明为什么现有结构不够
2. 再修改黄金切片
3. 最后把变化推广到其它资源

不能一边生成代码，一边默默改变结构。
