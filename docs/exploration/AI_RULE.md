# Archetype Admin AI 约束

## 目标

这份文档用于约束 AI 在本项目里的生成行为。

重点不是让 AI “更聪明”，而是让 AI 在稳定的边界内工作，避免生成兼容性分支、重复结构和不一致目录。

## 当前角色定义

AI 在本项目中的角色是：

- 结构执行者
- 规则遵循者
- 样板复制者

不是自由发挥的架构师。

## 必须遵守的规则

1. 新功能必须优先放入既定目录结构中，不能擅自发明新的顶层组织方式。
2. 页面组件不能同时承担类型定义、接口映射和展示逻辑。
3. 资源字段变更时，先修改该资源下的 schema，再修复引用处。
4. 严禁使用 `foo || oldFoo` 这类兼容写法保留旧字段。
5. 严禁在页面中直接写 `fetch`、`axios` 或随意拼接接口逻辑。
6. UI 必须优先使用已有的 `shadcn/ui` 组件。
7. 如果需要新增组件，必须记录新增命令和原因。
8. 能通过既有脚本、模板或生成器稳定完成的内容，优先走脚本，不要让 AI 每次重新空写。
9. 对现有结构有疑问时，优先参考仓库文档，而不是自行猜测。

## 类型与 schema 规则

### 必须做的事

- 优先在 `features/<resource>/schema/` 下定义类型
- 区分实体结构、表单结构、查询结构
- 让页面、映射、hooks 都依赖 schema

### 不允许做的事

- 不要在页面文件里直接定义大段业务 interface
- 不要把所有资源都塞进一个全局 `schema.ts`
- 不要在多个文件里重复定义同一个资源字段结构

## 页面规则

页面应该尽量薄，只负责装配：

- hooks
- 资源级组件
- 通用 UI 组件

不应该在页面中完成：

- 后端字段到前端字段的转换
- 大量本地状态管理来替代 Refine hook
- 业务类型的临时定义

## Data Provider 规则

`dataProvider` 负责数据协议，不负责承载全部业务复杂度。

当前原则：

- 应用级 provider 放在 `app/providers` 或现有 provider 目录
- 资源级字段映射放在 `features/<resource>/api/mapper.ts`
- 如果后端结构和前端结构不一致，优先在 mapper 层处理

## 变更规则

当需求变化时，采用下面顺序：

1. 改需求拆解结果或 contract
2. 改 schema
3. 改 mapper
4. 重新生成脚本可覆盖部分
5. 改页面和组件
6. 跑验证
7. 补文档

不要直接从页面开始补丁式修改。

## Script-First 规则

以下内容只要可以稳定模板化，就应优先脚本化：

- contract 初始化骨架
- 资源目录骨架
- TypeScript 类型文件
- mock 数据文件
- CRUD 页面骨架
- 表格列和表单字段的初始配置

AI 适合处理的是：

- 需求差异理解
- 特殊业务逻辑补齐
- 文档与规则收敛
- 在既定结构内完成非模板部分

AI 不应反复承担“从零手写标准骨架”这类低杠杆工作。

## 当前建议的 AI 输入方式

以后让 AI 生成页面或资源时，至少同时给它这些上下文：

1. 当前资源名称
2. 当前资源所在目录
3. 资源 schema 文件
4. 当前项目的结构文档
5. 当前项目的 AI 约束文档
6. 最近的黄金切片样板

如果缺少这些上下文，AI 很容易重新发明结构。

## 工具绑定说明

如果未来要写 `.cursorrules`、Codex skill 或其它 AI 工具规则文件，这份文档应作为上游约束来源。

也就是说：

- 工具规则可以变
- 仓库内的项目规则不应依赖某一个具体 AI 工具

这样后续切换工具时，不需要重写整套方法论。
