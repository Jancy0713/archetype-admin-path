# Archetype Admin 探索记录规则

## 目标

这份文档用于约束探索阶段的记录方式，确保后续可以把零散尝试沉淀为开源项目的方法论，而不是只留下口头结论。

## 记录原则

1. 只记录真实执行过的内容。猜测、计划、建议必须单独标注，不能伪装成执行结果。
2. 先记录命令和结果，再总结经验。避免只写“做了什么”，不写“怎么做的”。
3. 如果官方工具链已经变化，必须同时记录：
   - 最初参考的命令或方案
   - 实际可执行的替代命令或方案
   - 差异原因
4. 每次新增基础依赖、CLI、UI 组件、脚手架选项，都要留下最小可复现信息。
5. 每个步骤至少要有一次验证动作，例如 `build`、`lint`、启动、截图或人工验收结论。

## 每一步的固定结构

每个探索步骤必须包含以下字段：

- `Step`：步骤编号和名称
- `日期`
- `目标`
- `输入背景`
- `执行结果`
- `关键选择`
- `执行命令`
- `实际偏差`
- `产出物`
- `验证`
- `后续补充`

## 组件记录规则

当引入 `shadcn/ui` 或其它组件体系时，必须额外记录：

1. 使用的 CLI 命令
2. 新增的组件名称
3. 为什么现在需要这些组件
4. 是否有命令废弃、交互变化或配置前置条件
5. 后续如果新增组件，要在最近相关步骤文档里追加“补充记录”，而不是只改代码不记账

## 适用范围

这套规则适用于：

- 框架选型
- 脚手架初始化
- 认证方案探索
- Data Provider 方案探索
- UI 组件体系接入
- Prompt / Harness / Rule / Skill 的设计与迭代

## 后续执行要求

后续每完成一个探索动作，按下面顺序处理：

1. 先执行
2. 再验证
3. 最后补文档

如果动作未完成，也要记录失败原因和阻塞点。
