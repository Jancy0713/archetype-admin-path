# 开发上下文原则

这个文件用于沉淀“会持续影响后续开发方向”的用户共识。

记录原则：

- 只保留会持续影响设计和实现方式的共识
- 重点记录流程原则、结构原则、协作边界和 handoff 原则
- 不展开某一轮具体业务需求
- 不把已完成事项当作原则重复记录

## 当前有效原则

**总体与方法原则**
1. 优先服务“前后端分离项目”的标准化交付，先做通用流程，不提前绑定具体行业表达。
2. 初始化阶段应先理解项目级前提和系统基座，不要一上来钻具体功能细节。

**流程与确认原则**
3. 流程顺序必须稳定：先分析，再澄清，再确认，再继续推进；不能跳过补充提问直接往下写。
4. Human Confirmation Gate 必须显式存在，不能隐式处理，也不能在流程里被弱化成一句带过。
5. 对话层只呈现真正影响下一步推进的确认项，不机械复述整份 Markdown 或 YAML。
6. 提问与确认体验应尽量结构化，优先给推荐值、候选项、简短解释和可直接回复的编号格式，减少开放式填空。
7. 候选项不是为了走流程凑数；原始材料已经明确的结论，应直接采用推荐值，不重复追问。
8. 所有需要人工确认的内容，都应尽量收敛到统一、可编号、可脚本处理的结构中，避免同一问题在多处重复表达。

**结构与隔离原则**
9. 数据层与阅读层必须分离：结构化产物优先用 YAML 承载，Markdown 主要作为人工阅读视图，由脚本稳定渲染。
10. 展示层可以按优先级分组，但数据层应尽量避免为同一类确认信息维护多套并行结构。
11. 文档和目录应持续收束，按流程阶段组织入口，避免提示词、规则、reviewer 和步骤资料再次堆成大杂烩。
12. `runs` 目录作为历史执行记录，在修复生成流程脚本、排查 Bug 或整体架构 Review 时，禁止去检查或修改 `runs` 目录下的内容。除非用户明确指定“审查本次生成的产物”，否则一律默认排除对历史 `runs` 记录的检索和修改。

**Reviewer 与脚本边界**
13. reviewer 必须由独立 reviewer 子 agent 或独立新上下文执行，主 agent 不能自己 review 自己，也不能换个口吻自审。
14. reviewer 不能只做泛审查；每个阶段都应有明确专项检查点，并限制最大 retry，避免流程失控或反复磨洋工。
15. 脚本负责结构约束、骨架初始化、校验、渲染和流程边界控制，不能代替 AI 完成语义收敛和内容判断。

**Handoff 与主链流转原则**
16. 上下游 handoff 只保留当前执行真正需要的最小约束，不把已明确延后的内容继续带入正式产物。
17. 一旦某项内容在人工确认中已明确“不做 / 本轮不做 / 延后讨论”，后续正式产物不应再次把它包装成待实现范围。
18. 相邻流程（如 `init`、`prd`、`contract`）之间，输入边界应尽量清晰，避免把上游过程性描述直接混入下游正式输入。
19. `final_prd -> contract` 的正式主链接点，应优先挂在 `prd-04` review 通过后的主链收口动作上，而不是再新增一套平行人工入口。
20. `final_prd -> contract` 应被视为独立正式步骤，且设计必须为“后续修改”预留回入口。
21. `final_prd` 后面应显式存在 `05 Contract Handoff` 这一步；它负责把一份 PRD 拆成多个独立 contract flows，给出它们的顺序、依赖关系和层级交接说明。
22. 一份 `final_prd` 可以产出多个 contract flows；一个 contract flow 的正式终点应收口为一个独立 `swagger/openapi`。
23. 主链解释优先级必须以 `init -> prd -> contract_handoff -> contract -> openapi/swagger -> develop` 为准。
24. `contract` 流程收口到稳定的后端 API 合同产物（`openapi/swagger`），不把前端实现试错塞进 `contract`。
25. 前端代码生成、实现试错和返工应从稳定的 `contract + openapi` 输入出发，作为后续 `develop` 流程处理。
26. 进入 `contract` 后应隔离原 PRD run：PRD run 是上游事实来源，下游只能读取，不能回写它的进度、状态或过程产物。
27. 每个 contract run/flow 只维护自己的执行状态；可以检查上游依赖的 release 状态，但不能替上游改状态。

**外部能力边界**
28. 外部 skill 可以作为辅助参考能力接入，但不能替代本仓库自己的结构协议、校验、渲染和 handoff 边界。
29. 引入外部 skill 时，优先看是否真实可安装、可复现、可稳定接入当前工具链，而不是只看理念或展示效果。
